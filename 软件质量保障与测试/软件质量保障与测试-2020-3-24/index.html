软件质量保障与测试--第四章 软件测试质量保障-软件测试基本过程

发表于 2020-03-24

软件测试在软件生命期的位置

  1. 软件开发的过程/瀑布模型：分析 -> 设计 -> 实现&调试 -> 测试 ->运行
    • 在瀑布模型中，软件测试被认为是在软件开发过程的最后阶段进行的”一次性(one-time)”活动
  2. 软件开发的过程/V-模型
    • 用户需求 -> 系统设计 -> 概要设计 -> 详细设计 -> 编码
    •        |                    |                    |                    |               |
    • 验收测试 <- 系统测试 <- 集成测试 <- 组件测试 <–
    • 测试级别相应的开发级别
    • 局限：测试活动从编码之后开始，违反尽早进行测试的原则
    • 组件测试 <=> 单元测试
    • 是针对软件代码的测试，称为狭义的软件测试
    • V模型指明了开发过程活动与测试活动之间的关系，也向测试人员指明了应该以何种顺序执行这组测试活动
  3. 软件开发的过程/W-模型
    • 测试对象不仅仅包括程序，还包括需求设计等
    • 相比V模型有利于更早地发现软件开发过程中的缺陷，降低维护的成本
    • 与V模型类似，W模型中测试仍然保持着严格的顺序关系，无法实现测试活动的迭代，无法很好地对之前的制品进行相应的修改性工作

基本测试过程

  1. 基本测试过程

测试计划（test planning）

  1. 识别测试任务，定义测试目标以及为了实观测试目标和任务确定必要的测试活动
  2. 是做好软件测试的前提，良好的、切实可行的测试计划，在测试过程中严格执行，以测试目标为导向进行测试工作
  3. 测试计划（test planning）（IEEE 829）
    • 标识测试项（对测试项的了解）
    • 需要测试的特征（对产品的了解）
    • 不需要测试的特征（对产品的了解）
    • 采用的测试方法（测试策略/测试重点和等级）
    • 测试项通过的标准（出口准则，度量）
    • 测试暂停准则和回复要求（非正常退出）
    • 测试完成所需提交的交付项（文档等）
    • 测试任务
    • 测试环境的要求（工具、资源）
    • 职责（资源）
    • 人员配备和所需要的训练（资源）
    • 测试的进度（日程安排和粗略的时间表）
    • 风险和应急措施
  4. 测试计划可以是同步进行的，在软件开发过程中就可以进行软件测试的规划

测试控制（test contro）

  1. 根据收集和报告的测试信息和度量而来取的指导纠正措施
  2. 包括：
    • 利用事先定义的度量来评估和分析测试结果
    • 监督和记录的度量可以衡量：
      • 测试进度
      • 测试的覆盖状况（覆盖度）
      • 测试的出口准则（exit criteria）
      • 测试过程是否偏离计划
    • 确定是否需要采取纠正措施
    • 确定是否需要更改原计划（计划偏离实际）
    • 对测试计划的执行要进行反债
    • 风险的持续监控

测试分析&设计（test analysis and design）

  1. 测试分析和设计是基于确认的测试目标将概活的测试目标转化为具体的测试条件（test condition）和测试用例（test case）的一系列活动
    • 测试条件 <=> 测试需求，指的是能够被一个或多个测试用例测试的条目或者需求
  2. 包括：
    • 评审测试依据（test basis）（需求、系统架构、设计和接口说明）
    • 评估测试依据和测试对象的可测试性（testability）
    • 通过对测试项、规格说明、测试对象行为和结构的分析，识别测试条件并确定其优先级
    • 设计测试用例（从逻辑测试用例/概要测试用例到实际测试用例/具体测试用例）并确定优先级
    • 确定测试条件和测试用例所需的必要的测试数据
    • 设计测试环境（test environment），包括必要的基本设施和工具等
    • 细化时间表

测试实现（implementation）

  1. 通过特定的顺序组织测试用例来完成测试规程和即本的设计，并且包括测试执行必需的在何其他的信息，以及测试环境的搭建
  2. 包括：
    • 从测试条件具体到可执行的测试用例（即从逻辑测试用例到实际的测试用例（test case））以及必需的测试件（testware（文档和工具等）
    • 借助于所谓的测试准则（test oracle），确定期望值/期望的反应
      • 可以是文档、程序，甚至人工验证
    • 根据风险确定测试用例的优先级
    • 生成并准备好测试数据（test data）*.
      • 一个测试对象的输入值和状态值以及期望值或在运行有关的测试用例后的期望结果（exoected result）或实际结果（actual result）
      • 真实际结果是在测试后才能获得的
    • 组合成测试套件（case suite）/测试序列（多个单个测试用例形成的逻辑集）并为一个尽可能有效的测试的运行实现测试规程（test procedure）
      • 用于被测组件/系统的一组测试用例。在这些测试用例中，一个测试的后置条件（postconditio）通常用作下个测试的前置条件（orecondition。
    • 准备测试用具和设计自动化测试脚本
    • 确定已经正确搭建了测试环境，以确保配置的正确性

测试执行（test execution）

  1. 测试某个软件组合或柔统并产生实际的结果的过程称为测试执行
  2. 包括：
    • 检查执行测试所需的前置条件是否准备好：
      • 测试环境包括测试对象是否按照规定构建并准备完毕
      • 测试程序，测试脚本等是否准备完毕
      • 另外，是否定义了测试系统管理员并做好了准备
      • 事件管理系统等是否准备就绪
    • 严格按照测试计划和测试规约说明执行手动测试的测试用例或自动化测试的测试用例
    • 比较实际的和期望的结果
    • 分析期望和实际结果偏差的原因（测试对象的错误，测试件（testware）的错误等）
    • 提交所有事件（incidents），即所有在一个测试过程中出现的，并且在此后还需要检查或关注的事件[IEEE 1008]
    • &&测试日志**（test log）是一个关于测试执行的按时间顺序的详细记录[IEEE829]，记录（对每一个测试用例）：
      • 测试的日期、时间&测试人员
      • 输入测试数据和期望结果
      • 测试对象的ID标识/版本号、测试工具和测试方法
      • 测试结果和后续动作（例如，错误报告，测试用例的修改等）
      • 有可能的话，给出错误的原因
      • 记录特殊情况
    • 报告发现的错误（偏差/错误记录）并尽可能帮助开发人员分析错误原因
    • 如有必要，补充新的测试
    • 执行确认测试（confirmation testing）再测试（re-
      testing）&回归测试（regression testine，用于确认错误更改是有效的；或者执行已经更改的测试用例。

测试评估（test eva/uation）& 测试报告（test reporting）

  1. 评估出口准则是将测试的执行结果和已经定义的测试目标进行比较的活动。这个活动在各个测试级别上都需要进行
  2. 包括：
    • 对测试记录/日志进行评估，以判断测试是否已经达到了根据测试计划规定的测试出口准则（test exit criteria）
    • 评价是否需要继续执行进一步的更多的测试或者应该更改已经定义好的&&测试出口准则**（例如，准则根本没法实现或者在有限的资源（费用，时间等）条件下没法实现）
    • 为利益相关者/相关决策人员提交测试报告（包括测试活动和测试结果的文档，和在规定的测试出口准则下的测试运行的评估[IEEE829]）

测试结束工作（test closure activities）

  1. 测试结束阶最的活动就是从已完成的测试活动中收美和整合有用的数据，这些数据可以是测试经验，测试件，影响测试的因素和其他数据
  2. 包括：
    • 检查，是否所有计划的交付物（deliverables）都产生并交付了？或产生并交付了多少？
    • &&事件报告**（incident report）是否关闭
    • 对未关闭的事件报告提交变更需求
    • 系统的验收文档状态
    • 记录和归档&&测试件**（testware）（测试规约说明.数据.测试日志（test log）等等）。
    • 保存测试环境和测试基础设备，以备以后可用同样的数据和在同样的环境下重复一个测试或为将来得项目使用（→配置管理（configuration management）。
    • 测试件（testware）交付给专门的管理/维护部门
    • 分析和整理从所有测试阶段中获得的测试经验和教训，为以后新版本和新项目的测试过程的改进提供帮助：
      • 分析每个测试活动的计划和实施之间的偏差，并判断原因
      • 考虑测试结果与测试开销的关系
      • 有关风险的经验
  3. 在如下情况下需要测试结束工作：
    • 当一个软件产品正式应用
    • 当开发（或测试）达到一个里程碑（milestone）
    • 一个维护版本结束时

测试过程改进TMMi

  1. TMMi成熟度模型：
    • 1级 初始：混沌的、不明确的过程
    • 2级 已管理：明确的，和调试分开
    • 3级 已定义：组织级的测试过程集
    • 4级 已测量：可测量的过程
    • 5级 优化：具备持续过程改进的能力

TMMi2级过程域（PA:Process Area）：

  1. 测试方针与策略
  2. 测试计划
  3. 测试监督与控制
  4. 测试设计与执行
  5. 测试环境
  6. 目标是验证产品满足需求，大部分利益相关者还是认为测试是在编码之后的过程

TMMi3级过程域（PA:Process Area）：

  1. 测试组织
  2. 测试培训方案
  3. 测试生命周期与集成
  4. 非功能测试
  5. 同行评审
  6. 不再局限于编码之后，而是完全集成到软件开发过程中和里程碑中
  7. 基础是组织级的测试过程集，明确定义，在测试过程中不断改进

TMMi4级过程域（PA:Process Area）：

  1. 测试测量
  2. 产品质量评估
  3. 高级评审
  4. 完全定义的、且具备良好基础的可测量过程

TMMi5级过程域（PA:Process Area）：

  1. 预防缺陷
  2. 质量控制
    • 基于抽样的质量检测与控制
  3. 测试过程优化
    • 微调，不断改进

课堂讨论

  1. 课堂问题1：你理想或设想中的测试过程应该是如何的？
  2. 课堂问题2：测试的入口准则与出口准则分别指什么？请用一个示例来解释
    • 入口准则：满足什么条件，可以开始执行测试
      • 开发人员已经开发完成，并且可以执行
    • 出口准则：满足什么条件，可以结束测试
      • 测试覆盖率
      • 测试用例通过率
      • 对每个功能的测试用例的数量
  3. 课堂问题3：如何指定测试对象或测试用例的优先级？
    • 可以利用需求的优先级来测试：卡诺模型中的三种需求
    • 被调用次数最多的方法或模块
    • 缺陷的严重程度
    • 不同测试阶段的优先级：功能、性能、安全
      • 看团队计划
      • 常规来说：功能优于非功能
    • 针对不同类型的测试项
      • 基于风险做优先级的评估
      • 风险 = 影响 × 发生概率
      • 根据客户需求
      • 根据成本做测试，成本高的优先测试
  4. 课堂问题4：有哪些测试执行的方式？
    • 人工测试
    • 自动化测试
      • 构造测试实例、执行程序、分析结果
      • 工具：selenium、codecov、Jenkins等
