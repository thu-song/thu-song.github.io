系统程序设计--第四节

发表于 2020-03-10

标准IO

为什么我们需要标准I/O库

  1. 减少系统调用的次数，而不改变用户的逻辑、习惯
    • 先把数据缓存到用户空间中，再从用户空间拷贝
    • 建造一个库，维护一个在用户空间的缓冲区

标准I/O

  1. 最核心的做法就是把数据缓冲在用户空间
  2. 减少系统调用的次数
  3. FILE 结构
    • 记录了和流描述符对应的fd，缓冲区和缓冲区的刷新机制
  4. 缓冲机制
    • Fully buffered (disk)
      • 对于用户缓冲区里的内容，用户调用fread读取缓冲区的内容，这个用户缓冲区对用户是不可见的
      • 之后是文件系统缓冲区，在kernel中
      • 最后才是磁盘(物理设备)
      • 当用户缓冲区的内容完全被读取，标准I/O库会启动一次系统调用从kernel的缓冲区中读取内容(通常为4K)
      • 4K –> 1次系统调用
    • Line buffered (terminal)
      • 键盘和shell交互
      • 在shell中输入，回车符才会把缓冲区的内容输送给shell
      • 输出时也是一样，遇到回车符才会刷新出去
    • Unbuffered (stderr)
      • 所做的输入和输出，跳过缓冲，直接输送给文件系统
  5. 调用fread是没有系统调用的：
    • 【提问】当用户缓冲区的内容完全被读取出来的时候，标准I/O库会启动一次从内核缓冲区的读写，这个启动时通过fread调用的吗？

architecture

  1. 用户程序 –(fread)–> 用户缓冲区(setbuf) –(read)–> 内核缓冲区 –(页面调度)–> 物理设备(stdin…)
  2. 用户程序 –(fwrite)–> 用户缓冲区(setbuf) –(fflush/write)–> 内核缓冲区 –(fsync)–> 物理设备(stdout…)

change the buf: setbuf() / setvbuf()

  1. 用fopen打开文件时，系统自动分配了一个缓冲区
  2. setbuf中buf长度至少为BUFSIZ(通常为4K)，因为系统通常按照BUFSIZ的大小来使用这个缓冲区

fopen()

  1. 当fopen打开路径时，缺省用到了标准I/O库中的缓冲机制
  2. freopen打开pathname路径连接到已经存在的数据结构的fp
    • 【提问】不是很懂是什么意思？
  3. fdopen对文件描述符操作
    • 链接了文件描述符和标准I/O流
    • 对socket、管道和网络通信操作，没有具体的路径，只有文件描述符

read and write a stream

fgets()

  1. gets()可能会引起缓冲区溢出
  2. fgets()有长度n，限制了缓冲区溢出

fputs()

  1. fputs()将以null终止的字符串写入指定的流，最后的’\0’不被写入
  2. puts()最后会将换行符写入标准输出，fputs()不写换行符

error handling

标准IO的效率

  1. getc()本身只需要很少的时间，因为从用户缓冲区读取内容，所以系统调用比较少
  2. getc()和fgetc()的时间差别在于实现方式：使用宏实现时间比较少
    • 【提问】为什么不统一使用宏来实现呢？

fwrite()不能跨平台

  1. fread和fwrite必须在相同的系统上

    • 不同系统上的结构成员的offset可能会不同
    • 多字节整数为二进制形式可能不同

  2. eg：

    • fwrite()将多字节的文件类型写入文件系统时，只是将内存空间中对浮点数组的表示直接写入
      • 如果不同系统的表示方式不同时，不能读到相同的数据

临时文件

  1. 如果ptr是NULL，那么生成的路径名被存在静态区域，返回一个指向这个地址的指针
  2. 生成的临时文件没有名字，存进文件的内容，之后转存到数据库等，这个文件就不需要了

tempnam()

  1. 如果定义了环境变量TMPDIR，它将用作目录
  2. 如果directory不为NULL，则将其用作目录。
  3. <stdio.h>中的字符串P_tmpdir用作目录。
  4. 本地目录（通常为/tmp）用作目录。

标准IO的替代产品：

  1. sfio
  2. Alloc Stream Interface
    • mmap
  3. uClibc C library

标准IO有什么缺点：

  1. C标准库的流是支持定位请求的，是全双工的，读和写是共用的同一个缓冲区，而标准IO库的读写大部分都是操作缓冲区，达到一定条件后才会与内核交涉(即减少了内核调用的次数)，所以，如果相连的读写两个操作之间没有协调(调用冲刷、定位之类的函数)，就会导致缓冲区使用冲突，如果一定要满足这种需求能正常工作，那么缓冲区的管理、流位置与内核中文件位置的同步等都会变得非常复杂。而事实上标准C库的实现并不会考虑也没必要考虑这些不规矩的操作，所以要遵守以上限制。读完之后做好准备再写，写完之后做好准备再读
