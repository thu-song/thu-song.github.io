  1. 2006年11月8日，时任微软开发部联席总裁的吉姆·阿尔软终于宣布，Windows Vista的RTM版完成，至此，Windows Vista长达5年半的马拉松式开发终于结束了，11月16日，Windows Vista面向MSDN和TechNet订户提供下载，次年1月30日，Windows Vista正式面向消费者发售。
  2. 糟糕的口碑和市场表现
    • 憋了5年多的大招，Windows Vista终于和消费者见面了。然而随着这款操作系统走向市场，微软并没有迎来期待中的鲜花和掌声，而是抱怨，铺天盖地的抱怨。
  3. Windows Vista较上一个版本WindowsXP增加了上百种新功能，其中包括被称为“Aero”的全新图形用户界面、关机特效、加强后的搜索功能（Windows Indexing Service）、新的媒体创作工具（例如Windows DVD Maker）以及重新设计的网络、音频、输出（打印）和显示子系统。Vista也使用点对点技术（Peer-to-peer）提升了计算机系统在家庭网络中的显示通信能力，将让在不同计算机或装置之间分享文件与多媒体内容变得更简单。

  1. 对需求的把握是产品成功的首要因素

软件需求分析内容

  1. 需求分析的具体内容，主要归纳为四个方面：

软件的功能需求

  1. 描述软件的各种可能条件下，对所有可能输入的数据信息，应完成哪些具体功能，产生什么样的输出
  2. 举例：微信支持朋友圈分享、微信支持语音转文字

软件的非功能性需求

  1. 软件性能指标，可靠性、安全性、容限等功能以外的需求
  2. 举例：微信群支持最大500成员、升级数据不丢失
软件需求分析的非功能性需求：性能需求
  1. 2009年，淘宝第一届双十一购物狂欢节，这年成交额为5000万。这是那个时代很少的购物平台能达到的高度。
  2. 2017年，淘宝双十一页面系统出现错误，很多买家无法购物或者购物失败。
  3. 2018年，2135亿元的购物没崩溃，但是退款崩溃了。最后天猫紧急出退款公告。
  4. 2019年，“双十一”物流订单达到12.92亿，1分钟，超65亿！1分36秒，超100亿！5分25秒，超300亿！
  5. 2019年，“我的退款还没提交，快递已经收到啦~”。用户的需求催生快递物流行业向数字化、智能化应用的转变
软件需求分析的非功能性需求：数据安全
  1. 2018年，Facebook信息泄露事件，Facebook市值蒸发360亿美元。
  2. 剑桥分析公司通过分析用户们的行为模式、性格特征、价值观取向、成长经历后，有针对性地推送信息和竞选广告，以影响美国选民在竞选中的投票。
  3. Facebook客户信息泄露事件已经演变成重大的政治事件、经济事件、金融事件、科技事件、大数据事件、客户信息保护事件等多重重大的、深层次问题。
界面需求——拟物化与扁平化之争
  1. IOS 7 设计扁平化：苹果要革自己的命

软件与硬件或其他外部系统接口

  1. 包括：人机接口：说明输入、输出的内容、屏幕安排、格式等要求

软件设计和实现上的限制

  1. 如软件运行环境的限制（选择计算机类型，使用配置，操作系统的限制等）、设计工具的限制（使用语言、执行的标准）和保密要求等
  2. 举例：支持Android4.x及以上的内核版本、接口后向兼容
设计约束
  1. 首先是Windows Vista对硬件的需求远远大于当时消费者手中PC配置的平均水平
  2. Windows Vista的最低硬件需求是800MHz以上主频的处理器、512MB的运行内存、支持Direct×9.0的显卡、32MB的显存和容量至少20GB的硬盘，建议配置则是1GHz以上主频的处理器、1GB以上的运行内存、支持DirectX9.0b和WDDM1.0的显卡、128MB的显存和容量超过128GB的硬盘
  3. 在2006年的当时，很多用户桌面上的电脑都还是512MB的运行内存，这仅仅满足了Windows Vista的最低硬件需求，在这种硬件条件下，Windows Vista不仅很难流畅地全特效运行，其系统运行速度、文件操作性能和游戏性能在同等的硬件配置下甚至还赶不上WindowsXP。除此之外，Windows Vista还面临软件臃肿和续航表现（对于笔记本电脑而言）不佳的问题

第二章 软件需求分析过程

大型企业要重视需求

  1. 某公司的客户导向
    • “以客户的需求为目标，以新的技术手段去实现客户的需求，技术只是一个工具”

如何识别真正的客户需求

  1. 客户需求 ≠ 客户要求
  2. “需要更快的马” –> 汽车(真正的需求是速度更快)
  3. “手机需要大电池”(续航更长)
  4. 客户需求不是一个或者几个客户说的话

大型企业要重视需求

  2. 真的客户需求，需要去粗取精，弃伪存真，由表及里，由此及彼，然后归纳出来的客户的真正需求是什么
  3. 需求收集的完备性和分析的深入、细致是是否能把握好需求的关键所在

需求分析目的和任务

  1. 目的：对用户的需求进行鉴别、综合和建模，清楚用户需求的模糊性、歧义性和不一致性，分析系统的数据要求，为原始问题及目标软件建立逻辑模型。
  2. 大型系统的需求，一般分为几个子系统。软件需求分析任务：
    • 确定用户目标和任务
    • 确定这些任务所支持的业务需求，并作为使用实例
    • 把任务描述成功能需求，是用户完成其任务；也可以描述成非功能需求，明确了系统的限制和用户对质星的期望
      • 非功能需求，很重要，往往不经意的一个细节，会对系统造成巨大的打击。

软件需求分析过程

  1. 软件需求分析是一个项目的开端，也是项目实施最重要的关键点。
  2. 软件需求分析开展步骤：
    • 定义项目的视图和范围
    • 确定用户类别，每个用户类中确定代表，使用实例进行开发，并设置优先级
    • 从用户那里收集质量属性的信息和其他非功能需求，详细拟定使用实例
    • 评审使用实例的描述和功能需求
    • 形成需求分析文档
  3. 软件需求分析内容应正确、完整、一致和可验证
  4. 软件需求表达无岔意性，具有可追踪性和可修改性
  5. 软件设计人员和软件测试人员不再需要需求方的接触

软件需求文档

  1. 需求文档在开发过程中起指导作用，是所有子系统项目规划、设计和编码的基础
  2. 软件开发团队启动前，需要针对软件需求分析文档，开展需求澄清
  3. 软件从测试团队，要从使用实例中开发出测试用例
  4. 软件需求分析文档，关键包括如下内容：
    章 描述
    用户需求定义 描述为用户提供的服务
    包括非功能需求
    可以使用客户可以理解的自然语言、图形或者其他表示法
    系统体系结构 描述所预计的系统体系结构的高层概览，显示各个系统模块上功能分布
    系统需求规格说明 更详细描述功能性需求和非功能性需求。还可以定义与其他系统的接口
    系统界面说明 包括图形化的系统模型，显示系统构件之间以及系统及其环境之间的关系

软件需求跟踪管理

  1. 平均水平的项目，平均有25%的需求变更——Jones 2000
  2. 需求变更导致的返工，占所有返工工作量的75%~85%—Wiegers 2003
  3. 需求管理，关注管理一组不断演化的需求：
    • 要接受“需求会变动”的事实，分析需求是要确定哪些是稳定的需求，哪些是易变的需求
    • 需求变更要经过评审，以一种可控制的方式将需求变更融入到项目中
    • 每项需求都能与其对应的设计、源代码、测试用例联系起来，并跟踪

第三章 软件需求分析实战：在线点餐系统

  1. 为解决现场就餐的各个难题，某大学准备开发一套在线订餐系统，覆盖大学内的所有的餐饮供应商，系统只面向大学生和教师，用户使用订餐系统时需实名注册
    • 订餐用户通过系统可以完成订餐和评价
    • 商家可以管理系统本商家的餐饮信息
    • 管理员可以管理用户和商家信息

软件项目设计约束

  1. 产品范围和用户群体
    • 产品范围：校网的在线网站
    • 用户群体：大学生和老师、校园商家
  2. 运行环境
    • 本系统主要采用以下技术，属于业界主流的成熟技术，且所开发系统业界已有成熟案例
    • 开发语言：Java
    • web 框架：Spring boot、vuejs
    • 数据库：MysQL
  3. 项目开发平台
    • 操作系统：Windows10
    • 应用服务器：Ubuntu Server 14.04LTS64位
    • 服务器配置：CPU：4核 内存：8GB 公网带宽：10Mbps 硬盘：100G

软件需求分析开展步骤

  1. 用户需求分析
  2. 使用Use Case方法分析功能性需求
  3. 非功能性需求范围分析
  4. 非功能性需求详细分析
  5. 输出软件需求规格说明书

第一步： 用户需求分析

  1. 店铺管理员
    • 能够在所管理的店铺中添加新的菜品或饮品
    • 能够在所管理的店铺中下架（删除）新的菜品或饮品
    • 能够在所管理的店铺中接受用户提交的订单
  2. 顾客（订餐用户）
    • 能够查看所有的店铺列表
    • 能够进入某个店铺内部查看具体的菜品/饮品
    • 能够在某个店铺内把指定的菜品/饮品加入到购物车
  3. 系统管理员
    • 审核店铺注册信息的真实性
    • 清除无效用户/店铺信息
    • 发布/修改/删除系统公告

第三步：非功能性需求范围分析

  1. 非功能性需求是指那些不直接关系到软件向用户提供的具体功能/服务的一类需求，它们通常与软件的总体特 性相关，如界面、安全性、响应时间、可靠性等
  2. 系统UI
    • 好的Ul设计不仅是让软件变得有个性有品味，还让软件的操作变得舒适、简单、自由，充分体现软件的定位和特点
  3. 安全性
    • 包括访问安全、数据安全等。访问安全要能应对黑客常用的SQl注入、OS命令注入、XML注入等攻击方式；数据库建立主从备份机制，主数据库遇到故障系统能自动切换到备份数据库
  4. 性能要求
    • 主要指用户高峰期登录系统后操作界面时的响应时间，系统需能支撑500个用户同时在线点餐时，在网络正常的情况下，界面响应时间要小于1S
  5. 可靠性
    • 在各种复杂的网络情况下，系统能够保持7*24服务不中断，用户随时可以订餐、进行店铺管理或系统管理

第四步

系统UI
  1. UI是系统中用户能直接看到并与之交互作用的部分，直接关系到用户体验的好坏，因此设计 一个美观、简洁、操作方便的用户界面非常重要
  2. UI框架
    • 由于前段框架使用vue，因此我们选择Element作为Ul框架，Element是套基于Vue2，0的桌面端组件库
  3. 浏览器支持
    • Web 浏览器：Chrome、IE11、Firefox、Opera、afari、及任何支持HTML5标准的浏览器
  4. 分辨率支持
    • 标准分辨率：19201080、16801050、1600900、14401050、1440*900、2K
安全性
  1. 由于在线点餐系统因为涉及到在线支付等敏感功能，因此对安全性要求较高
  2. 软件安全性（Security）是指为了保持信息的机密性、完整性和可用性，系统对恶意威胁的防护能力。可能的威胁包括但不 限于未授权访问、使用、泄露、破坏、修改、毁灭等
  3. 本系统应能抵御如下几种常见攻击：SQL注入、XSS (Cross-Site Scripting，跨站脚本攻击)、OS命令注入攻击等
可靠性
  1. 软件可靠性是软件产品在规定的条件下和规定的时间区间完成规定功能的能力。软件可靠性不但与软件存在的缺陷有关，而且与系统的输入和系统的使用有关
  2. 为了让在线点餐系统给用户提供高可靠性的服务，对标业界经验，并结合系统每年允许出现的故障次数和故障等级，对系统可靠性标准做了如下定义：
    序号 故障描述 允许理次数(/年) 故障等级
    1 用户使用在线点餐系统无响应或发生系统错误，导致用户无法正常点餐、支付，店铺无法接单等，且在24小时内没有解决问题 0 严重
    2 用户在线点餐、提交订单、在线支付、在线接单等响应极慢，且该异常状态持续30分钟以上，且12小时内没有解决问题 低于2次 重大
    3 独立页面出现故障，未对其它业务流程造成影响，且在12小时内解决问题 低于10次 可控
系统性能
  1. 在线点餐系统的用户主要是校园师生，使用特点是早中晚的用餐时间为系统的使用高峰期，结合这些特点，我们重点关注：系统的平均响应时间和并发数
    • 平均响应时间：是指从发送请求到收到请求返回结果的平均时间，直观上反映系统的快慢，因此需要对重点业务要求有较短的响应时间
    • 并发数：是系统同时能处理的请求数，例如，在晚饭高峰期，系统要能支撑50个用户同时提交订单
  2. 在线点餐系统平均响应时间要求：
    序号 业务 平均响应时间
    1 用户：浏览餐品、点餐、提交订单页面
    商家：接单、查看订单页面    		 1-2s
    2 用户：查看订单状态、对订单/餐品评价
    商家：查看餐品剩余数量、用户评价等2-3s
    3 商家：查看月度营业额等统计数据
    系统管理员：查看平台各个店铺营业额    		 3-5s
  3. 在线点餐系统并发数要求：
    并发数 说明
    □ 100<并发数 能支撑小于100人同时点餐
    √ 100并发数<1000 能支撑大于100人但小于1000人同时点餐
    □ 1000≤并发数 能支撑超过1000人同时点餐

课堂讨论

  1. 企业微信的文件大小限制过低，仅有10M；将移动端的需求与PC端混为一谈，100M文件大小限制足以应对移动端无线传输能力，但对PC端而言100M文件的本地传输是远远不足的；

    • 腾讯本身的考虑：个人生活和工作分开

  2. 还有聊天框删除会导致聊天记录丢失。不知道是否是安全性的需求才导致了这样的设计

    • 处于隐私的考虑，如果支持聊天框删除，说明在服务器存了用户的聊天记录

  3. 给人的感觉就是微信在努力做一个轻量的社交软件，但是事实上他的用途又是工作和学习

    • 腾讯方面一直不想让他作为工作学习工具

  4. 拓展性不够，用户的需求增加，微信不能很好满足

  5. 运行LOG记录在开发时这个需求一般是接纳的。

    • 状态监控：可以监控系统的运行状态。
    • 安全审计：通过对日志进行分析，可以发现是否存在非授权的操作
    • 问题追踪：发现程序BUG，安装配置的使得问题
    • 但软件发布给用户时，日志需要加密保护

  6. 我觉得“运行log记录”这个需求是需要接纳的，因为log记录的作用一方面可以显示程序当前的运行状态，另一方面可以在程序出现问题后定位问题。通过对日志记录的统计分析，发现和排除出错原因，也可以加强系统的维护和管理
