系统程序设计--第三节(file and directory)

发表于 2020-03-08

stat()、fstat()、lstat()

  1. 文件状态的访问函数

struct stat

file type

  1. regular file
  2. directory file
    • 不能被通常的read、write读写
  3. character special file
    • 设备文件，如/dev/tty0
    • 串口、键盘等
  4. block special file
    • 硬盘、U盘、SSD
    • 磁盘，代表一个分区，名字类似于/dev/hda0
    • 对于Linux来说，所有的设备在文件系统中都有一个对应的文件
    • special file没有内容，只有名字
    • 当进行读写的时候，kernel会先确定是一个special file，读写时调用驱动，转换成设备驱动程序的读写，如hdread，hdread会和设备进行交互
      • 所谓交互是将一些指令放进设备的IO的内存空间中
      • 磁盘上一部分内存芯片做数据缓冲，一部分内存芯片做指令空间
      • 磁盘和OS之间有协议，比如格式，写入那个内存地址
    • read读磁盘数据的过程：
      • kernel会确定读的是设备文件
      • 调用驱动，将read转换为驱动程序的read
      • 这个read会和设备进行交互，把指令放进设备的IO的内存空间
      • 磁盘会解析指令，读取相应柱面、磁道、扇区的数据，放进磁盘的数据缓冲区
      • 驱动从磁盘的数据缓冲区读文件，搬到主内存对应的位置(文件系统的缓冲区)
      • 再搬到read的目的buf中
  5. FIFO
    • 管道，和字符文件很像，但是要保证先进先出
  6. socket
  7. symbolic link
    • 符号链接
    • 打开符号链接文件的时候，OS会识别是符号链接文件，真正的用意是打开所指向的真正文件

user ID of process

  1. 每一个文件都有一些ID
  2. 进入shell以后，启动一个进程，比如执行 $ ./a.out，OS会为这个进程指定一个 user ID 和 group ID
  3. ID 名称 代表含义
    real user id

    real group id    		 who we really are
    启动者的ID，是运行这个进程的用户ID，此时的进程和启动者有一样的权限，进程只是用户的工具
    effective user id

    effective group id

    supplementary group id    		 used for file access permission checks
    一般情况下，也是代表启动者的ID
    saved set-user-id

    saved set-group-id    		 saved by exec functions
  4. 进程权限：real user id判断的是进程的权限
  5. 进程对文件权限：每次访问一个文件，都是用effective user id进行权限匹配(owner group others)
  6. 一般情况下：real user id / real group id = effective user id / effective group id = (logined user) user id / group id
  7. effective user(group) ID决定了进程对文件的访问权限

set-user-id

  1. real user id 和 effective user id 可以不同，有以下几种改变方式：
    • root可以改变进程的euid，但是一般没必要更改，本身已经是root权限
    • set-user-id
  2. 文件的访问权限中，有一个bit为 set-user-ID
    • 这个bit通常用于可执行文件
    • 如果文件的该bit被设置，任何运行这个文件的进程的effective id 将会被设置为文件的 owner id
      • effective user ID <– owner id of the file
      • effective group ID <– group owner of the file
  3. eg：/usr/bin/passwd 访问 /etc/shadow
    • 这是一个可执行性文件
    • 它的set-user-ID bit被设置
    • 普通用户执行时，对应进程的 euid = 0(root)
    • 那么passwd可以访问shadow的内容，也可以修改
    • 但是paaawd的代码限定了只能修改shadow中属于自身的哪行：会先确定real user id，根据这个判断shadow中的某行密码
    • 相当于私有数据，只能通过方法passwd修改，passwd限定了修改范围
  4. 实验：st_uid
    • 两个文件：myread和1.txt
      • -rwx–x–x 1 song fudan /home/song/myread
      • -rw——- 1 song fudan /home/song/1.txt
    • 设置setuid之前
      • 用户song运行myread可以读取1.txt
      • 任何其他用户运行myread都不能读取1.txt
    • 设置setuid之后：chmod +s -read
      • -rws–x–x 1 song fudan /home/song/myread
      • 所有用户都能运行myread来读、写1.txt
    • 总结
      • 实现对特定数据文件指定访问方法

文件访问权限

st_mode 屏蔽 意义
S_IRUSR 用户-读 0400
S_IWUSR 用户-写 0200
S_IXUSR 用户-执行 0100
S_IRGRP 组-读
S_IWGRP 组-写
S_IXGRP 组-执行
S_IROTH 其他-读
S_IWOTH 其他-写
S_IXOTH 其他-执行

目录的执行权限

  1. 如果目录没有可执行权限，那么我们就不能从该路径搜索包含的内容
  2. eg：/usr/bin/ls
    • 如果bin没有可执行权限，那么不能从bin执行ls
  3. 目录的可执行权限是用于做搜索的
  4. 实验：open /home/song/class/1.txt
    • 如果路径上任何一个目录(/、home、song、class)没有可执行权限，那么我们无法解析到1.txt文件
    • 如果想要创建或删除一个文件，那么需要父目录的写权限和可执行权限

文件访问权限

  1. 从EUID和EGID判断文件访问权限
  2. 对文件访问权限的判断顺序：
    • root (无需判断)
    • user (文件访问权限的owner部分)
    • group
    • others
  3. 只要判断出来一次，就不会继续判断
    • 比如是文件的owner，没有任何权限，那么就不会继续判断
    • 即使所在的group有读写权限

新建文件和目录的owner

  1. 当我们creat一个新的文件
    • 文件的 user/group ID = 进程的 effective user/group ID
    • 什么意思？the group ID of the parent directory(set-group-ID for the parent dir)
  2. 【提问】PPT中提到的创建新文件时的组ID会跟随①进程组effective ID②父文件夹ID 那么会不会出现这两个矛盾的情况呢？这时候怎么确定组ID呢

access()

  1. 利用进程的real ID来确认是否有权限
  2. access用的是real ID，而 open用的是effective ID

umask()

  1. 如果设置了umask，那么在umask之后的creat文件的权限会被mask
    • 即便是creat时设置了相应权限，也会被mask掉
    • 缺省为0000

chmod()

  1. 改变访问权限
  2. 当我们想要更改访问权限时，通常的做法是：

粘住位

  1. 如果一个可执行程序文件的这一位被设置了，那么在该程序第一次执行并结束时，该程序正文的代码被保存在交换区(swap区）
    • (程序的正文部分是机器指令部分)
    • 这使得下次执行该程序时能较快地将其装入内存区
    • swap区相当于是硬盘和内存进行快速交换的地方，相当于硬盘的缓存，虚拟内存的一部分
  2. 如果我们试图设置普通文件的粘住位(S_ISVTX)，而且又没有超级用户优先权，那么mode中的粘住位自动被关闭
  3. 现在用于保护不同用户之间的文件（可读写，但不能删除）

chown()

  1. 改变文件的owner
  2. 这个方法是有限制的：
    • BSD限制只允许root改变owner
    • POSIX允许用户改变自己文件的owner，但当设置 _POSIX_CHOWN_RESTRICTED
      • 只有root可以改变一个文件的owner
      • 一个非超级用户可以改变文件的group ID
      • a) 进程拥有此文件
      • b) 可以更改你所拥有的文件的组ID，但只能改到你所属于的组
  3. 之所以有限制是因为
    • 我们需要对文件进行统计
    • 或者对每个文件的owner都有不同的权限，黑客可以改变做坏事

File Size

  1. 对于普通文件、目录文件和符号链接的文件大小，多少个字节
  2. 符号链接文件大小，大小是link本身的size
  3. 文件空洞

truncate()

  1. 截取文件的长度，截断到length长度
  2. 也可以截取到比原来文件长的长度，XSI-conforming systems会增大文件的大小
    • 这个有可能会造成维护的困难（为什么？heap也可以增大分配空间的大小，类似的机制不可以吗）

link 和 unlink

  1. 每一个i-node都有(hard)link count，多个link对应到多个name
  2. 符号链接
  3. 为了防止产生环路，会禁止对目录创建链接
  4. 最后一个unlink = rm删除
  5. 只要文件被打开，就不会被删除，当文件被关闭，kernel会检查count
  6. 如果pathname是符号链接，那么unlink会删除这个符号链接，被引用的文件不会被删除

Linux特性：可以unlink正在打开的文件

  1. 巧妙应用：如果在www服务器上发布，可以通过url访问
    • 如果想要修改文件错误，unlink之后增加一个新文件
    • 不支持该特性的话：假如文件频繁被访问，那么文件就不能被删除
    • 在Linux中，unlink之后，过段时间，所有打开被关闭之后，就会被删除，此时可以上传新文件

rename()

  1. Linux系统中，需要先增加一个newname的link，之后把oldname unlink掉
  2. 如果旧名称是目录，如果存在新名称，则必须引用目录，并且该目录必须为空
  3. 如果旧名称或新名称都引用了符号链接，则该链接本身将被处理，而不是指向的文件

符号链接

  1. 跟随符号链接：access、chdir、chmod、creat、exec、link、mkdir
  2. 不跟随符号链接：chown、lchown、lstat
  3. 符号链接也会产生闭环
    • 系统会提示：ftw returned -1: Too many levels of symbolic

symlink() / readlink()

utime()

  1. 记录了从1970年1月1日开始到调用时刻的秒数
  2. 如果times为空指针，则访问时间和修改时间都设置为当前时间
    • 为此，要么进程的EUID必须等于文件的owner ID
    • 要么进程必须对该文件具有写许可权
  3. 如果times为非空指针，则将访问时间和修改时间设置为times指向的结构中的值。
  4. 我们无法指定状态更改时间st_ctime的值。

mkdir() / rmdir()

  1. mkdir中mode_t至少一个可执行位被设置，允许访问目录中的文件名

read directories

  1. 所有有访问权限的用户都可以读目录，但是只有kernel可以向目录写入（疑问？）
    • 牵涉到目录的实现，这个写操作不是在目录中创建一个文件的写操作，而是对目录文件的写操作
    • 什么是目录文件呢？home是一个目录，是有内容，home在文件系统中是怎么实现
    • 必须通过kernel给的方法，比如creat来写目录，这个数据不是私有的，可以读，但是不能写，delete，unlink，来产生后果，这些系统调用，通过kernel对目录文件的inode进行操作
  2. readdir每次只可以读一个entry

chdir() / fchdir() / getcwd()

  1. 程序里面存储的是当前目录的inode

special file / device file

  1. 每个文件系统都有其主、次设备号
  2. 系统中每个文件的st_dev值是文件系统的设备号，表示在那里，在哪一个分区上
  3. 只有字符特殊文件和块特殊文件才有st_rdev值，此值包含该实际设备的设备号，表示原始的设备文件
  4. 主设备号的意思是属于哪一类：0类、1类、2类、…
    • 所有的键盘是同一类的、所有的磁盘是同一类的、所有的串口是同一类的
  5. 次设备号是设备在整个系统中的id
    • 测设备号不同代表在不同的分区上(不同的位置)
  6. ls -l [设备文件]
    • 只会列出设备的rdev(raw device number)
  7. root下面的文件如果挂载在root上，那么它的dev number和磁盘相同
    • 如果是特殊设备文件，那么还有rdev

sync(void)

  1. 有的系统会把sync放在定时事件中，定时启动运行
  2. 将缓冲区的内容同步到磁盘中
  3. 是一个启动的动作
    • 把脏数据的块号放在队列中准备写回
    • 返回时，不代表动作完成
