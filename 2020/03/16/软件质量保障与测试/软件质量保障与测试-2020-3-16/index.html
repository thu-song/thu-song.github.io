软件质量保障与测试--第三章 软件测试质量保障-需求文档与建模技术

发表于 2020-03-16

模型的概念

  1. 模型是已存在的现实或将要被创建的现实的抽象表现
  2. 模型特性：
    • 表示特性：模型映射现实
    • 归纳特性：模型化简和归纳所代表的的现实
    • 使用特性：模型为特定的目的而构建

模型语言

  1. 句法：使用符号或符号组合
  2. 符号：模型基础元素
  3. 使用模型语言的先进之处：
    • 减少歧义；清晰
      • 相比自然语言描述
    • 集中在需求的某一个角度
      • 数据、功能、行为角度
  4. 模型语言与自然语言相结合
    • 总体的模型描绘总体的架构
    • 细节用自然语言补充

目标模型(Goal Models)

  1. 目标是利益相关人所描述的，关于系统的目的、属性或者使用的意图
  2. 系统愿景定义了系统的最高目标，其它目标都是对系统愿景的精华
  3. 记录利益相关人的目标是需求分析最为基本的工作
  4. 可以使用自然语言来描述系统目标
  5. 可以使用模型语言来描述系统目标“与或树”（AND/OR tree）
    • 一个AND/OR目标树由代表目标的结点和代表目标间AND分解与0R分解关系的有向边组成
    • 每一个结点（除根结点外）仅与一个父目标相关联

使用与或树(AND/OR Tree)

  2. OR分解表示只要有一个子目标实现了，就认为父目标实现了

  3. AND分解表示只有所有的子目标被实现，才认为父目标实现了

目标描述的七个原则(扩展知识)

  1. 简明扼要地描述目标
  2. 使用主动语态描述目标
  3. 准确描述利益相关人意图
  4. 分解高层目标
    • AND/OR分解
  5. 说明目标所创造的价值
  6. 描述引入目标的原因
  7. 在目标定义中避免不必要的约束
    • A require B，A目标要实现的话，必须首先实现B目标

用例/用况

  1. 用例/用况（Use Case）
    • 描述可能在施动者和系统之间发生的交互，当系统执行时则提供附加价值
    • 从用户（或其它外部施动者）的角度来看，用例对系统做规格说明：
      • 每一个用例都描述一些系统必须向用例中所涉及的施动者提供的功能。
  2. 包括两个部分：
    • 用例图
    • 用例规格说明书

使用UML建模语言的用例图

  1. 统一建模语言：UWL(Unified Modeling Language)(OMG 2007)
  2. UML是一个通用的标准建模语言，可以对任何具有静态结构和动态行为的系统进行建模
  3. UML的目标是以面向对象图的方式来描述任何类型的系统，具有很宽的应用领域
    • 其中最常用的是建立软件系统的模型
    • 但它同样可以用于描述非软件领域的系统，如机械系统、企业机构或业务过程，以及处理复杂数据的信息系统、具有实时要求的工业系统或工业过程等
  4. 类图
    • 静态关系，在整个系统的生命周期中都是有效的，不会发生变化的
  5. 活动图
    • 一个活动到另一个活动的流程，包括系统运行中并发的活动
  6. 用例图
    • 用况和参与者进行关联
    • 从用户的角度描绘系统静态使用情况
  7. 状态图
    • 由状态转换、事件和活动组成的
    • 描述类的对象的所有可能的状态以及事件发生时的转移条件
    • 可以认为是对类图的补充，需要为那些有多个状态的、行为随外部环境改变的类画状态图
  8. 此外，还有对象图、交互图、组件图，部署图等

UML用例图基本元素

  2. 用例图（UseCaseDiagram），展现了一组用例、参与者（actor）以及它们之间的关系，它们之间的关系构成的用于描述系统功能的静态视图

    • 用例（use case）
      • 可以有延伸要求，比如导航到目的地要求避免阻塞
      • 延伸要求可以不画，在规格说明中描述
    • 参与者（actor）
      • 人或者第三方系统
      • 系统边界之外
    • 系统边界（system boundaries）
    • 扩展关系（extend relation）
      • 用况B中有些占位符，使用用况A中特定的流程填充占位符
      • 【疑问】
    • 包含关系（include relation）
      • 用况A的执行过程包含了用况B的执行流程
    • 参与者与用例之间的关系(relation between actors and use cases)
      • 提供数据
      • 得到反馈结果

需求的三个视角

基于数据视角的数据建模

  1. 实体联系图（Entity-Relationship Diagram）
    • Peter Chen（1976年提出），它在数据库设计领域得到了广泛的认同
  2. 统一建模语言-类图（UML Class Diagram）

实体联系图

  1. 实体联系图（E-R图）：实体、关系、属性三个基本概念概括数据的基本结构，从而描述静态数据结构的概念模式。

  2. 实体：具有相同属性的实体具有相同的特征和性质，用实体名及其属性名集合来抽象和刻画同类实体。例如，乘客；消费者；路径；导航仪。

  3. 属性：实体所具有的某一特性，一个实体可由若干个属性来刻画；例如：乘客这一实体，可以有“姓”，“名”，“ID”，“预定坐位”

  4. 联系：联系也称关系，信息世界中反映实体内部或实体之间的联系和联系的特征。实体内部的联系通常是指组成实体的各属性之间的联系；实体之间的联系通常是指不同实体之间的联系。同时在无向边旁标上联系的类型（1：1，1：n或m:n）

基于功能视角的数据建模

  1. 数据流图（Data Flow Diagram）
    • 需求的功能视角要处理的是把从环境中接收到的输入数据转换成释放到系统环境中的输出数据
    • 通常情况下，例如在汤姆·戴马克（Tom DeMarco）所著的“结构化分析”（Structured Analysis）一书中，把数据流图用作功能模型
    • 系统和系统环境的图形表示称为环境图（也称上下文图，context diagram），在特定情况下，如果数据流图是用来定义系统边界的话，数据流图也称为环境图
    • 由于在数据流图中没有控制流程，或者比如说进程的内部工作方式并没有显示出来，因此，数据流图要用附加的、结构化形式的说明加以补充

活动图

  1. 活动图(UML Activity Diagram)

    • 从功能的角度，活动图适合对系统活动的顺序进行建模
    • 活动图模拟活动节点和活动节点之间的控制流
    • 对象流表示控制流的一种特殊形式。活动图中的同步条允许模拟并发控制和对象流
    • 可以使用判定节点来描述备选控制(alternative control)和对象流

基于行为视角的数据建模

  1. 在需求建模中，系统的动态行为是从行为的角度来建模的
    • 从这个角度来看，重点在于能够发现一个系统的各种状态以及造成状态变化的事件
  2. 动态描述系统行为
  3. 状态图包括系统的状态、状态间的转换、及触发状态转换的事件。
  4. 一种是基于Mealy 1955年及Moore 1956年的状态图
  5. 一种是UML State Diagrams（OMG2007）。它吸收到前一种的模式，并得到了加强。

总结

  1. 模型语言可以避免自然语言的缺点；
  2. 模型语言可以专注于某一个方面的需求；
  3. 模型语言应与自然语言联合使用；
  4. 了解各种建模的适用范围，及各个元素和模型所要表达的内容。
