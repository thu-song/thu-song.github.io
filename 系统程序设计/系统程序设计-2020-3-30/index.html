系统程序设计--第七节(process relationships)

发表于 2020-03-30

用户登录

  1. 操作系统启动之后，初始的代码的进程ID=0
    • 在传统Linux中会变成swap，做内存交换的进程
    • 同时，也会启动init进程
  2. init进程，ID=1
    • init读配置文件，启动其他进程，比如网络服务、终端服务
    • 重要作用是负责用户登录
    • 在最开始的时候，linux系统装在小型机上，供很多人同时使用计算机服务
    • 一台主机放在后台，每一个用户都有自己的终端
    • 每一个用户都可以通过终端登录到后台服务器上
  3. 登录过程：
    • init进程启动之后可以从配置或者硬件检测获得主机有多少个终端
    • 针对每一个终端，都可以启动为这个终端服务的一系列进程
    • init会先为每一个终端fork一个终端进程
    • init进程在每一个中断进程中exec一个进程getty，初始化终端设备和环境设置
      • 得到终端的参数配置，相当于把自己的0、1、2映射到这个终端，准备读终端的输入(等待用户输入用户名和密码)
    • 输入用户名之后，就可以检测用户名是否在用户配置文件中
    • getty的重要作用就是调用login，把加密后的密码和配置文件中的用户数据进程匹配
      execle(“/usr/bin/login”, “login”, “-p”, username, (char *) 0, envp);
  4. login得到登录用户名后：
    • 得到用户名之后就去比较用户名是否正确
    • getpwnam() —— 读取passwd文件
    • getpass()，接着读用户键入的口令
    • 登录成功之后
      • 启动缺省的shell进程
      • 把shell进程的ID，real UID等都设置好
      • 把登录的进程终端关联到shell上
      • shell只是后台的进程，进程的0、1、2指向了前端能看到的shell(伪)终端，之前是物理的屏幕和键盘
    • shell
      • 打印message-of-the-day
      • 检查新邮件
    • shell开始运行，父进程ID是init进程ID = 1
    • 登录shell读其启动文件，准备接收命令
  5. 老式系统中，shell进程连接到物理终端上
    • 如果退出之后，重新回到getty状态，等待新的用户输入
    • 退出之后，shell终止

课堂笔记

  1. 有没有进程没有终端(没有输入输出)？
    • init进程
    • 服务器进程不需要终端，甚至应该被禁止，因为ctrl-C会终止
  2. 我们通常打开的shell命令行是终端
  3. 为什么设置进程组是由shell来做的，因为当运行的时候，并没有声明加入那个进程
    • shell看到命令行，fork子进程，先设置进程组，再exec
    • exec后，进程组ID会被继承
    • 在输入输出重定向时，也是shell先fork子进程，再exec
  4. /bin/shell或者更改登录shell
