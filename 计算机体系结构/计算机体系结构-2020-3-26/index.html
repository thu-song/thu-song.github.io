计算机体系结构--第四节(多周期处理器：datapath and control)

发表于 2020-03-26

1. 处理器设计的过程

  1. 自顶向下：
    • 分析指令的执行语义
    • 找到执行语义需要的组件
  2. 自底向上：
    • 基于现有技术制造组件
    • 将指令需要的组件拼在一起，形成可工作的电路
    • 从中发现有用的信息 – 最长路径、最短路径、形成关键的时序
  3. 反复改进，给出一个部分解，不停扩充和改进得到可工作的处理器
  4. 单周期处理器中是通过控制指令(microinstruction)来实现的，通过控制信号控制组件一起协同完成指令的执行
    • 控制器就是一种有限的自动机，PC作为状态，通过PC获得更详细的信息
    • microinstruction也可以控制指令的序列

2. 设置多周期处理器的设计通道

  1. 计算机的五种组件：
    • 控制
    • 数据通路
    • 内存
    • 输入
    • 输出
  2. 单周期处理器设计过程的不足：
    • 指令执行的时间不同
      • 关键路径就是load指令
      • 所有指令都需要在一个时钟周期内完成，导致指令执行的时间很长
    • 真实的内存并不是理想化的
      • 真实的内存并不能在一个时钟周期内完成读和写

1.内存访问时间对处理器设计的影响

  1. 容量越大，访问时间越慢
    • 解决：层级存储
    • 把不同容量和不同速度放在一起：L1和L2cache直接放在处理器中，内存昂在处理器外部，通过总线访问
    • 降低处理器访问数据的时间

2.如何降低处理器的时钟周期

  1. 给定一个复杂的组合电路，将组合电路一分为二，每一部分构成独立的组成电路
  2. 两个组成电路中引入存储单元隔开，原来一个大的组成电路需要在一个时钟周期内执行所有功能
  3. 现在把分在两个不同的组成电路，执行相同的功能，需要两个组成电路按照时间顺序完成
  4. 这样，电路变小，执行更快：原来在一个慢的时钟周期内执行功能分解成两个快的组成电路的功能
  5. 通过这种划分，把组合电路的电路变短，并且可能导致一些指令中的组件可以被移除掉，让组合电路更加简单

3. 关键路径

  1. 理想的寄存器和内存：
    • CLK输入仅仅在写操作时有影响

关键路径的五个因素

  1. 取指
  2. 译码
  3. ALU计算地址
  4. 访问内存
  5. 写回寄存器中

4.理想的内存和真实的内存

  1. 理想的内存
    • 写入的时候需要有效的时钟信号
    • 访问内存的地址、写入的数据以及写使能必须在setup阶段变得稳定
  2. 真实的情况
    • 寄存器和内存没有时钟
      • 因为写比较慢
    • 也就是说，写操作必须是一个带时延的组合电路
      • 写使能变成1，写入的数据已经准备好
      • 写过程是时延的过程
      • 写操作之后，数据写入内存相应地址
    • 写使能时，数据和地址必须是稳定的，才能保证写成功

5.race condition

  1. 在单周期处理器中，如果没有时钟，那么寄存器和内存不能可靠的工作
    • 对寄存器而言，在写使能变成1时，寄存器的地址并不一定能够稳定
    • 寄存器的地址和寄存器的写使能可能存在竞争
      • 先后顺序不同会导致写入的情况不一样
  2. 内存也有同样的情况

如何避免race condition

  1. 多周期处理器的解决方案：
    • 关键：地址要变得稳定
    • 假定在第n个周期时，地址稳定
    • 在n+1个周期，写使能信号变为1
    • 在写使能信号变成0之前，地址信号不可以改变，可以保证写成功
  2. 两个访问端口的内存：
    • 读内存和写内存两个端口独立
    • 在同一个时钟周期中，读和写可以访问不同的位置
  3. 读端口是一个组合电路：
    • 读地址有效
    • 经过时延
    • 数据输出有效
  4. 写端口也是一个组合电路：
    • 写使能=1
    • 经过时延
    • 数据会写入对应位置

6.时钟周期的基本的限制

  1. 计算下一个PC的逻辑电路
  2. 决定把指令从内存中读出来的时间
  3. 访问寄存器的时间
  4. ALU计算的时间

7.切分CPI=1的数据通路

  1. 把长的数据通路切分成小的部分
  2. 在每一部分之间加入寄存器

3. 处理器设计的步骤

  1. 在多周期处理器设计中，对多周期处理器进行了划分，引入了电路之间的寄存器，因此为了区分指令集中的寄存器和电路中的寄存器：
    • 指令集中的寄存器称为逻辑寄存器
    • 电路中的寄存器称为物理寄存器
  2. 处理器设计的五个步骤：
    • Step 1:ISA=>Logical Register Transfers
      • 指令 -> 寄存器变化的过程
    • Step 2:Components of the Datapath
      • 数据通路的组件，即组合电路
    • Step 3:RTL+Components =>Datapath
      • 寄存器状态变化逻辑+构造的组件 -> 组成数据通道
    • Step 4:Datapath+Logical RTs=>Physical RTs
      • 数据通道+逻辑寄存器变化 -> 物理寄存器
    • Step 5:Physical RTs=>Control
      • 生成控制信号

1. 第四步：数据通道+逻辑寄存器变化 -> 物理寄存器

2. 多周期处理器的数据通路

  1. 相较单周期处理器
    • 内存变成两个端口：写端口和读端口
    • 只需要一个加法单元：
      • 减少了PC+4和PC+SEx两个加法器
      • 只剩下了ALU算数单元
  2. 多周期的方式对电路划分之后，单周期中很多组件的访问冲突在多周期中不存在，把存在冲突的操作放在不同的周期，使组件的数量减少
    • 这是多周期设计带来的其中一个好处

3. 以branch指令为例

1. 取指

  1. 取指阶段，所有的指令执行相同的操作
  2. 在时钟刚开始时，PC寄存器中的值是当前指令存储的位置
    • 需要从内存中读出来，写到指令寄存器中
  3. 在指令读取结束时，在时钟周期结束，需要更新物理寄存器

2. 译码/取寄存器

  1. ALU没在使用
  2. 分支指令的取寄存器时，并没有使用ALU运算单元
    • 此时下一条指令的PC值已经计算出来了
    • 可以计算target的位置

3. branch指令执行完毕

4. 有限自动机

  1. 由两部分构成：
    • 状态
    • 告诉我们如何更改状态
  2. 如何更改是由下一个状态函数决定
  3. 除了我们必须明确的信号，有的信号状态可以为x
    • 要求要么置为1，要么置为0
    • x的也置为0

5. 控制模型

  1. 定义了我们怎么控制寄存器之间的转移
    • 控制信号构成了state
  2. 状态转移发生在当前状态退出时
    • 当前状态生成的控制信号，控制当前指令退出时，也就是时钟周期下降沿，需要计算下一个状态
    • 根据当前状态和输入值，可以计算出下一个状态的值，从而计算出下一个时钟周期时的控制信号
  3. 当前的状态+正在执行的指令+一些条件 -> 下一个状态+下一个状态生成的控制信号

6. 总结

  1. 对于单周期处理器
    • 时钟周期较长
  2. 多周期处理器
    • 一条指令分成更小的步骤
    • 每个时钟周期只执行其中的一步
  3. 把数据通路分成同等大小，使时钟周期变得更小
  4. 控制通过有限控制图定义
  5. 一些特殊的状态图可以用微序列器表示出来
  6. 控制设计规约为微编程
  7. 在复杂指令集和受限的数据通路中，控制会变得复杂
  8. 简单的指令集 + 强大的数据通路 -> 简单的控制

7. 异常

  1. 不受自己程序控制的转移
    • 记录触发异常的地址
    • 记录其他用来返回的信息
    • 异常处理前保存用户的状态，异常处理后恢复用户状态

1. 两种异常：中断和陷阱

  1. 中断：
    • 由外部事件导致
      • 网络、键盘、磁盘和时钟
    • 异步的发生
      • 大部分的中断都可以暂时被屏蔽
      • 电源的失效是不可屏蔽的
    • 这样的事件可以在指令间被处理
    • 暂停当前程序的执行，处理完之后再恢复
  2. 陷阱：
    • 由程序内部实现导致的
      • 溢出
      • error(内存访问)
      • fault(访问的页面不在内存中)
    • 同步的发生
    • 异常处理需要采取补救的措施
    • 可以尝试重新执行指令
      • 程序继续执行或者被终止

讨论

  1. 多周期用microcode，单周期用PLA，优缺点
    • microcode：
      • 修改更为方便，用计算机控制计算机
      • 对程序员更友好
      • 需要查表，如果是zero或者inc都会比较简单，但是load需要查表，就会比较复杂
    • PLA：
      • 如果设计出现问题，控制信号不易修改
      • 指令数较少时，每条指令有一组控制信号，组件少，控制信号少
        • PLA表不会很大，实现更简单，电路更简单，更快
      • 但是扩展性比较差，整张表会比较大，PLA实现会更复杂
      • 除了存储之外，其他的会比较简单
  2. 单周期需要exception处理吗？
    • x86会有一个program state，ZF/CF告诉发生overflow
    • 在MIPS中，也可以抛出异常来处理
    • 处理器模式的分页模式，可能会发生缺页
    • 单周期处理器早期功能比较简单，用在嵌入式领域
      • 指令数少
      • 数据都在内存中
      • 单周期中指令一定会执行结束，处理器简单
      • 处理器执行中假定内存中读出来的都是ok的等
      • 异常情况下，处理器崩溃，也无所谓
  3. 没有实现的指令如何处理？
    • 仿真，通过已有指令的组合来执行
      • 但是因为发生了exception，发生了上下文切换，执行较慢
    • 处理器厂商，在之前设计了一些处理器，设计较差
      • 把之前处理器的指令扔掉
      • 少部分指令不能跑，可以用模拟
      • 找新的等同的指令
      • 用新的指令组合来模拟
      • 向后兼容
    • undefined，硬件翻译成当前处理器的指令
      • 大部分时间都不实用
    • 现在的CPU跑的比较快，现在x86
      • 不用考虑硬件指令集，处理
    • 早期x86处理器上没有浮点计算
      • 生成的程序，如果编译器要生成浮点指令
      • 正常的情况下，程序并不能运行
      • 浮点和整数的只是不同的表示格式
      • 可以用软件去模拟，可以工作，但是程序执行很慢
    • 硬件的bug要软件来填
      • 硬件的bug会触发exception
      • 固件来改
      • 给操作系统厂商写补丁，以软件的方式对硬件的行为进行修正
      • 硬件的bug是不可修改的
