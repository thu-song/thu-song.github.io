操作系统Ⅰ--第二节(Chapter2：Operating-System Structures)

发表于 2020-03-18

回顾

  1. 桌面操作系统：通用计算机，通用操作系统，linux Windows
  2. 嵌入式：手机、pad等
    • 比一般的嵌入式复杂
    • 现在趋近于桌面，但是仍有差距，作为开发环境的主要还是桌面
  3. 操作系统：好用又不会滥用
    • 操作系统的功能越少，用户越自由；功能越多，对用户而言越傻瓜
  1. 操作系统的结构：局限于公开的部分
  2. 关注
    • 理解操作系统构建方式，以方便我们更好的使用
    • 作为开发者的角度，以什么方式看待它

操作系统服务

  1. 提供的功能以系统服务作为接口实现
  2. 用户接口：用户直接使用的称为用户接口，用户直接交互的方式使用
    • CLI(Command-Line)，无格式或少量格式修饰，本质上是字符流，ASCII格式修饰
      • 专业使用者的工具
    • Graphic User Interface(GUI)
      • 一开始也是为特殊的使用者，有专业的用途，比如图形设计
      • 普通使用者使用OS的便捷方式，以鼠标和桌面为核心
      • windows操作系统以GUI为主，辅以CLI
    • Batch
      • 不面向人
      • batch和shell的差别
      • 是一种机制，指操作系统可以接受不同的任务通过命令脚本以集中的方式完成
      • 虽然CLI也可以重定向到文件，但是CLI的本意就是从输入流中获取命令
      • 面向无接口的方式运行
      • 执行中不与用户交互
      • 输入输出不能在运行中以显式的方式展现给用户
  3. 程序执行：
    • 计算机的所有功能都是用过应用程序的执行体现出来
    • 操作系统中存储了程序，如何运行这些程序
      • CIL执行一个进程，Ctrl+C终止进程
      • GUI通过文件浏览器找到exe文件或者快捷方式来启动进程，打开进程任务管理器杀死
      • 在CLI中或者GUI找到与进程关联的数据文件，比如.docx文件，通过start ex.docx启动一个进程并且打开文件
        • 文件类型绑定的方式来启动一个进程
  4. IO操作
    • 直接通过操作系统执行IO操作
    • 内部命令：cp等修改文件系统内容
  5. 修改文件系统的内容甚至文件系统本身，管理操作系统静态的方面
  6. 通信：
    • 操作系统上的进程与相同操作系统的其他进程或者其他操作系统上的进程之间的通信
    • IO，或者OS提供的通信机制实现
    • 比如从数据库中得到数据，如何从sql server获得数据
    • 两个进程共享内存，share memory
      • 确保时间的同步，一个进程写完，另一个进程读
      • 速度快，提高大量数据通信的效率
      • 本质上仍然依赖于跨进程消息传递的机制实现——IPC
  7. 错误检测，OS需要检测进程运行中的问题，并且可能的修复
    • 可能发生在CPU、内存硬件、IO设备上
    • OS必须很好地修补错误
      • 可以重新执行操作，如果第二次执行成功，可以认为第一次错误没有发生，尤其对于随机性发生的错误
    • 有的错误无法修补，OS不能无视错误的出现，必须终止当前进程的执行
      • 不能持续无休止的重新执行
      • 也不能忽略错误
      • 与用户目的相关联的错误，不得已可以终止进程执行
        • 需要提供给用户足够的错误信息
        • 需要提供debug
  8. 另外的功能：
    • resource allocation
      • 提供给我们一个虚拟的环境，独自占用所有资源
      • 也可以运用这个功能定制OS为进程分配资源的方式
    • Accouting能力
      • 记录特定用户对特定资源的使用
      • Windows系统管理器中的事件(打印事件)
      • 打印服务，虚拟打印机可以设置权限，谁可以打印，谁不可以打印，如果需要支持网络打印服务，需要根据打印的量来收费
        • 一种方式在客户端做收费，先填表，提供打印任务
        • 如果有不需要填表的客户端，那么这个计费机制会失效
      • 所以需要在分配资源的一方记录计费
      • 记录各种能力，供事后分析
    • 对资源的保护与安全
      • 主要针对多用户的环境
      • 授权
      • 保护指提供一种机制使得任何对系统资源的访问是可控的
      • 安全是对系统外部的访问要求用户认证
  9. 操作系统核心——services：程序执行、IO、文件系统、通信等
    • 这些服务通过系统调用以标准的接口发布
    • 用户可以使用那些功能，由系统调用的接口决定，接口公开什么功能，用户用什么功能
      • 用户的进程可以直接使用系统调用
      • 系统调用有关部分可以以用户接口的方式直接提供给用户使用
        • 包装GUI组件，帮助用户以图形方式使用各种系统能力的子集
    • GUI、CLI、batch只能通过用户接口使用公开的能力

CLI

  1. 最常见的形式就是shell
  2. 本质上是一个字符流
    • 通过换行符，呈现二位的形式
    • 屏幕上可能包含多个字符流的内容
      • 标准输入流在屏幕上显示是不连续的(被标准输入隔开)，但是本质上是连续的
      • 除了用户以外的标准输出流、标准错误流

GUI

  1. 必须有键盘、鼠标和监视器、图标、folder(文件夹)
  2. 大多数现有的操作系统，通用操作系统通常同时提供CLI和GUI
    • 手机通常除非有root，才会找到CLI
  3. 特殊的GUI
    • 触屏：支持同时选中多个点
      • 模拟增强原有鼠标的能力：右键、一般的点击、双击等
      • 在鼠标不可用的情况下使用
      • 通过手势提供能力
      • 提供虚拟键盘，图形化的代替
    • 鼠标的局限：
      • 特定时刻只能指定一个点，而且速度有限
      • 选择多个点，可以使用圈选的方式，选定范围，得到包含和不包含的关系
      • 鼠标上只有两三个按键

系统调用

  1. 用户接口只会公开OS服务的一部分
  2. 所以对用户来说，更重要的是系统调用提供的服务
  3. 系统调用接口也称作API，主要面向的对象是进程、用户程序
    • 用户接口只是提供了API的一个自己
  4. API不完全等同于系统调用，API是系统调用的一个子集
    • 系统调用可能以不同的形式包装呈现
    • Windows下最完整的系统调用服务：Win32 API
    • POSIX API，是Linux操作系统公共的接口，使得不同操作系统上，代码不需要修改就可以执行
      • 是系统调用的一个自己
    • Java API是JVM的API接口
  5. 为什么使用API，而不直接使用系统调用
    • 有了API，头文件、链接库文件，就可以像直接使用用户库一样调用系统功能
    • 而不需要了解系统调用的细节
    • 系统调用在每个操作系统中不同
    • 只需要掌握高级语言的概念就可以使用系统调用的能力
  6. 系统调用序列
    • 得到输入文件名
    • 得到输出文件名
    • 通过系统服务打开源文件
    • 打开或者关闭目标文件
    • 通过循环，从源文件读取内容写入目标文件
    • 直到一次读操作失败，必须是表示EOF的失败
    • 关闭输出文件
    • 为什么没有关闭输入文件
      • 关闭输出文件是为了确保已经全部写入磁盘
      • 因为进程退出会自动关闭输入文件，因为只是读，不会对源文件产生实质性的影响
    • 结束当前进程

标准API

  1. 标准API的函数内容在操作系统内部
    • 只是包装了系统调用的细节
    • 很薄的一层
  2. 系统调用的细节由链接库实现
  3. API可以支持不定多个参数，比如printf
    • 但是系统调用必须指定参数个数，需要分配栈空间

系统调用的实现

  1. 系统调用如何将高级语言调用转换为系统内部的系统调用
  2. 每个系统调用与一个系统调用的号相关联
  3. 系统调用将高级语言传进来的号，产生一个中断
  4. 通过使用API调用系统调用
    • 用户只需要知道遵循API调用的规范，知道返回结果
    • 所有细节都封装在一个运行时支持库中
    • 可以作为compiler的一部分发布
      • 很短的一段代码，把输入参数用上，执行系统调用的位置，把返回结果打包返回
      • 操作系统运行时不需要这个库，都是静态链接
      • compiler支持某个操作系统，表示能打包系统的运行时库和头文件
  5. 所有系统调用通过统一的入口进入，通过号找到函数指针，跳转到函数位置
    • OS只需要将每个系统调用代码的第一行代码填入表格
    • 通过中断返回，```iret``，中断返回指令
    • 返回到用户态，链接库的代码中，再把返回值返回给API函数的调用者
    • 用户态：调用者、调用前的代码(系统中断的服务)、调用后的代码
    • 系统调用中间可以停下来，
    • 异步的情况下，多个线程，系统调用中断，当前线程的流被打断，直到系统调用函数返回才会执行下一次指令
    • 内核中的exit也会把当前进程销毁

系统调用的参数传递

  1. API的作用在调用前和后与用户提供的东西进行关联
  2. API拿到输入的参数，必须以某种方式在实现的时候被用到
  3. 有些参数可以通过通用寄存器来传递
  4. 用户态一个内存区域的首地址穿进去没有意义，地址本身的值只在进程的上下文中才有意义，是一个虚拟地址
    • kernel必须利用虚存的映射信息，将其转换为物理的值
    • 因为在虚存中连续的内存空间在物理上是不一定连续的
  5. 必须实现用户缓冲区的内容到内核缓冲区的拷贝，不能通过参数传递机制传进来
  6. 通过栈传递参数
    • kernel套无法访问到用户栈上的内容
    • 必须在栈上拷贝
    • 如果系统调用产生了结果要返回给用户，必须以相反的方式传回去
  7. 如果有多个线程访问参数
    • 必须抓取参数的静态内容，抓取一个快照
    • 发出调用时，要确保在参数拷贝完成前，其他线程不能对参数进行修改
  8. 参数传递方式：
    • 寄存器
    • 整个内存块以某种方式保护起来，做一个kernel态的拷贝
      • 必须考虑到内容在用户的原地址中被修改
      • 如何保证调用中参数内容的确定性
      • 其它进程和其他线程都有可能修改
      • 指向与其他进程的共享内存区域
    • 通过用户栈
      • 实现栈上局部内容的拷贝
      • 在kernel态创建一个临时的栈
      • 把用户栈栈顶的一部分内容拷贝到kernel栈
      • 更好的方式
    • 后两种方式都可以以不限的方式指定不定的参数和不定的参数量

系统调用的类型

  1. 程序控制：
    • 正常结束、异常终止
    • 加载、执行
    • 创建进程和结束进程
    • 得到进程属性、设置进程属性
    • 等待、休眠状态
    • 休眠直至特定的事件和信号发生
    • 分配和释放内存
  2. dump memory if error
    • 排错中分析
    • 内存的值
    • 将整个虚拟内存dump
  3. debugger
    • 以为着被调试程序的内容可以被调试程序修改
    • 断点以什么机制实现的，进程如何停在断点
      • 任意的行上停下来，唯一的方法是修改程序执行的代码段
      • int 3覆盖指令的第一个字节，将原来的指令保存
      • int 3执行之后，将原来的指令填进去
      • 这称为代码断点，和指令位置相关的断点
      • 在一个特定的系统调用上设置钩子，在特定的系统调用上停下来
      • 数据中断要用到虚存的能力，将变量所在的页设置为保护
        • 对变量的读和写都会产生异常
        • 只能在页层面上设置保护，无法精确到一个变量而不波及到其他变量
        • 再恢复页的保护，在页的后面位置设置临时断点
        • 重新执行产生异常的指令
        • 指令在临时断点停下
        • 恢复之前对页的保护
        • 取消临时断点
  4. 锁
    • 限制
  5. 文件管理相关
    • 创建、删除文件
    • 打开、关闭
    • 写、读
    • 得到、设置文件属性
  6. 设备管理相关
    • 请求设备、服务设备
    • 读、写
    • 得到设备属性、设置设备属性
  7. 信息维护
    • 尤其是系统信息维护
  8. 通信
    • 进程内、进程间通信
    • 通信往往带有服务的要求：一致性等
    • 传统：send，receive
    • 共享内存的方式
    • 容器：将共享内存限制在容器内部
  9. 保护：
    • 为不同的资源在访问时设置保护
    • 允许或者拒绝用户访问

标准库

  1. 实现C语言的通用性，语言的统一，可以移植
  2. 过于通用的也不好
    • 很少的标准库中会与设备打交道
  3. 绝大多数应用都或多或少的依赖操作系统
  4. 有真正的实现部分，与链接库不同
  5. 有的标准运行时库需要依赖操作系统完成
    • print打印到屏幕
    • 必须在不同操作系统上实现不同系统调用
  6. 运行时库是按照操作系统分的
  7. 既可以以静态库的方式存在，也可以以动态库的形式存在
  8. 纯内存操作不依赖与操作系统
