操作系统Ⅰ--第三节(接上节)

发表于 2020-03-25

Example：MS:DOS

  1. 服务于单任务，运行在单核的、单个CPU的计算机
    • 除了内核之外，整个操作系提供可以可以空闲，也可以是进程空间
    • kernel 启动 command interpreter，执行相应的进程
    • 进程启动时，command interpreter 部分会被释放
  2. 只有进程的加载概念，没有进程的创建部分
    • 没有数据结构去描述进程本身，即操作系统就是进程
  3. 单一的内存空间
    • 内存空间：从地址到存储单元的映射关系
    • DOS只需要一个物理地址空间
    • 所有资源都可以使用，甚至可以安全的覆盖或修改kernel 和 command interpreter，只要进程结束之后，可以返回至kernel
    • command interpreter根据内存需要，可能会被丢掉一些
      • 但是kernel不会被覆盖
  4. 程序exit不需要返回到进城本身，只需要将进程释放
    • 检查command interpreter是否完整，如果不完整，从磁盘上重新加载
    • 跳转至命令提示符

Example：FreeBSD

  1. 多任务
    • 必须有进程数据结构管理多个任务
  2. 系统启动之后，启动kernel部分
    • kernel启动完成后，执行user login
    • 根据用户的配置文件创建一个代表用户的shell进程
  3. shell 进程执行任何可以执行的命令行
    • 需要时可以调用fork函数，创建新的进程
    • 在新的进程中，通过exec创建不同于shell的内存映像
    • 根据前台执行或后台运行，可以需要wait到进程结束
  4. 新创建的进程同样可以退出，调用kernel中的exit
    • 由kernel释放进程资源
  5. 本质上和Linux相像，细节上略有不同

系统程序

  1. 系统程序本身不是操作系统kernel的部分，也是用户程序形成的
    • 帮助我们使用操作系统
  2. 绝大多数(普通)用户对操作系统的看法是由系统程序来定义的，而不是由系统调用来定义的
    • 大多数用户不会通过写代码来使用操作系统的功能
    • 系统服务往往通过系统程序来提供
    • 比如文件管理，不会为文件管理专门写程序来应用
  3. 系统程序的帮助下，可以比较方便的进行程序开发和执行程序，比如
    • 文件管理：操作系统静态的在磁盘上，动态的在内存中
      • 目录的管理
    • 状态信息：硬件配置、时间、日期、内存量、磁盘空间等
    • 文件内容修改
      • 文本编辑器
      • shell中修改命令
    • 对程序语言的支持
      • 编译器、汇编器、debugger、解释器
    • 程序加载和执行
      • 绝对定位、可变址的定位、链接相关、覆盖定位
    • 通信
    • 后台服务：
      • 后台意味着不会干扰控制台的运行，后台一般不和控制台打交道，一般通过管道、文件
      • 利用其他方式查看后台进程的状态，比如日志
      • 有的只运行一次，比如检查磁盘
      • 有的一旦启动，需要持续工作，特别是为硬件服务
      • kernel部分为硬件服务的称为driver，关注终端处理
      • 有的driver在用户环境中运行，比如关注侦听数据，服务于硬件，一直在后台运行直到关机
    • 应用程序
      • 可以伴随OS安装的程序称为系统程序
      • 但是有的系统程序和应用程序没有区别

OS的设计与实现

  1. 操作系统的设计与实现不是”solvable”，但是有些方法被证明是成功的
  2. OS本身是一个软件，所有有开发者
  3. OS的设计开始于目标和标准
  4. 受硬件的改变、系统的类型的影响
    • 可移植
    • 在相同的平台上，创建一个全新的操作系统
    • 新的需求带来新的操作系统设计：DOS->windows
    • 对shell的程序不满意
      • 重构
      • 知识产权的冲突
  5. 用户目标和系统目标
    • 用户目标-操作系统应该使用方便、容易掌握、可靠、安全、快速
    • 系统目标-操作系统应该易于设计、实现和维护，以及灵活(适应不同操作系统的使用)、可靠、无错、高效
  6. 区分：
    • policy：做什么->接口
    • mechanism：如何做->实现
    • 使得系统实现具有足够的灵活性
      • policy在未来可以修改
      • mechanism相应调整
  7. 高可用性和高可移植性使得操作系统移植变得简单

实现

  1. 开发操作系统必须存在可以不用运行时支持的语言
  2. 太多的变化
    • 早期使用汇编语言，高于机器语言
    • 然后是系统编程语言像Algol, PL/1
    • 现在是C, C++，写OS时，C++的功能受限制
      • 比如C++的多重继承，取决于内存状态，比如静态变量的初始化
      • 不可以在不明确操作系统的内存实现的情况下使用
  3. 实际上通常是多种语言的混合
    • 最底层——汇编，甚至机器语言
      • 跨平台跨硬件、设计
    • 主体用C
    • C、C++或脚本语言PERL、Python、shell 等脚本
  4. 更高级的语言更容易移植到其他硬件
    • 但是更慢
  5. 仿真(Emulation)可以允许操作系统在非本机硬件上运行
    • 在完全不同的硬件平台上运行特定平台上的特定操作系统

操作系统的结构

  1. 下载Linux kernel的源代码
  2. 代码量大，需要正确的结构
  3. MS:DOS，分层调度，
    • ROM中的BIOS服务驱动
    • MS-DOS服务驱动
      • 提供中断服务，提供大多数服务内容
    • 驻留系统程序
      • 驻留程序是一种特殊应用程序，它在装入内存运行后，其部分代码仍然驻留在内存，当该段代码被激活时，它又进入运行状态。常用的驻留程序是作为某个中断处理程序的一部分，其激活条件就是系统产生了此中断的中断请求
      • 比如自己写的中断程序，不能退出，需要一直在内存中
      • 也可以用来做多任务切换，不从内存中退出，而是驻留在内存中，等待被切换
    • 应用程序内容
  4. Unix：
    • 系统程序部分和kernel部分
    • kernel包括了各个服务的入口和相应实现
    • 提供文件管理、缓存管理等，文件管理和缓存管理相互影响，甚至相互依赖
  5. 分层方法：
    • 不同的运行层次
    • 不同的代码会运行在不同层上
    • 层之间的接口是明晰的
    • 0层是硬件，N层是UI，用户接口
    • 分层的前提是，函数只能调用低层次的功能
  6. 微内核系统结构
    • 尽可能减少操作系统的kernel部分，更多的移动到用户态
    • 例子：Mach
    • 典型的应用，苹果系统的kernel(Darwin)集成了Mach的功能
    • 每个功能的主要部分都在用户态，需要通过通信机制，来实现相互数据和功能上的服务
    • 优点：
      • 易扩展
      • 易移植
      • 可靠性，内核态的故障很难恢复到出错前的状态，用户态可以很简单的把用户进程杀掉重新尝试
      • 更加安全，在用户态执行的代码没有能力直接访问内核的数据结构，比如以内核态可控的方式来访问

模块

  1. 基本功能一般是静态的，在不同的操作系统上不同
  2. 对硬件的支持能力
    • 在特定的计算机系统上，硬件配置的可能性很少
    • 一般的操作系统为了向前兼容，操作系统比如满足不同配置的要求
  3. 创建可加载的kernel模块
    • 在启动时不会加载
    • 在系统启动后，可以动态的在kernel的内存空间中加载或移除某部分的代码
    • 启动以后，在用户的干预下，或者脚本的指导下，动态的加载

Mac OS X 结构

  1. 不大允许用户直接调用系统服务，只允许调用应用接口
  2. 可以开发一些应用，直接调用系统服务，但是通过企业发布

iOS

  1. 移动的操作系统
  2. 与Mac OS X在上层很想，应用容易移植
    • 不能直接运行Mac OS的应用程序
  3. 软件本身是架构中立的，无论底层是ARM还是Intel的CPU，都可以执行
    • Cocoa Touch：使用objective-C，C语言的扩展，面向对象
    • 视频音频的支持
    • 内核支持
  4. 内核基于Mac OS X内核

Android

  1. 底层是Linux，iOS的底层是BSD(Unix)
  2. 基于Linux核
  3. 运行环境，一般来说不允许用户直接在OS上开发
  4. Java的bytecode作为程序开发的底层的语言
    • Android应用程序和底层的处理器结构也分隔开
  5. 提供在手机上开发必须提供的功能
    • web浏览器(webkit)
    • 数据库(SQLite)
    • 多媒体
    • 小的libc，介绍耗电
  6. Android也可以在Linux上开发
    • native的开发，JNI库(Java native interface)，在上层也可以用Java开发

操作系统调试

  1. 调试是去除bug
  2. 操作系统本身可以有日志
  3. 一旦出现不可恢复的故障(从程序的角度来说不可恢复)
    • 生成core dump文件
  4. 操作系统调试性能
    • 前台的程序和后台的程序争夺资源
    • 前台CPU不足，卡顿
    • 可以减少后台的资源分配
  5. 同样的操作系统在不同硬件上的能力怎么样，可以作为操作系统能力的体现：
  6. profiling：
    • 除了崩溃之外，性能调优有时还可以使用活动的跟踪列表来优化系统性能，为分析分析而记录的是指令指针的定期抽样，以查找统计趋势
  7. Kernighan定律:“调试的难度是编写代码的难度的两倍。因此，如果你把代码写得尽可能的聪明，你就没有足够的智能来调试它。”

性能协调

  1. linux中的top，Windows中的任务管理器

DTrace

  1. 分析程序运行中产生的系统调用
  2. 反向工程的工具，更好的理解程序是怎么运行的

操作系统的生成

  1. 硬件配置非常多的情况下
  2. 桌面计算机可以直接下载镜像，底层架构种类很少
  3. 安卓只会在固定的设备上，除非现在USB，没有必要在操作系统中携带当前操作系统不需要的代码
    • 而通用计算机必须都有
    • 安卓必须根据实际配置，产生系统的实际内容
  4. 利用硬件系统的配置对操作系统修剪
    • 定制化产生平台独有的操作系统
  5. Linux自己编译自己的内核，把不需要的内核代码从系统中去掉
    • 这些模块可以保存在磁盘上，但是不会被加载到内存上
    • 定制kernel的原因：
      • 幻想去除不用的代码提高效率(代码并没有被执行，并没有提高效率)
      • 减少磁盘占用

系统启动

  1. 系统上电，硬件从没有能源状态进入能源激发状态
  2. CPU上电，决定下一条指令的硬件(PC)，初始化每次启动都是固定的值
    • 在有些处理器中是全0，比如ARM
    • 有点处理器由于历史原因有特定的值，比如x86
      • 任何x86兼容的处理器都在ffff上的地址
      • 启动之后进入实模式，跳转到fff0上
  3. 初始化的代码必须在每次启动的时候不变，位置和内容也不变
    • 固件ROM，存储启动代码
    • 称为bootstrap loader
    • 代码常常保存在ROM或者EEPROM中
    • 代码必须找到kernel代码，跳转到kernel代码
  4. 先找到boot block，再进一步找到kernel位置
    • 通过USB、网络、软驱都可以直接找到boot位置
